Kasus 1: Aplikasi Ojek Online (Tabel Driver)

"Halo, kami ingin membuat tabel untuk menyimpan data driver ojol kami. Setiap driver pasti punya Nama. Yang paling penting, Plat Nomor kendaraan mereka tidak boleh ada yang sama di sistem agar tidak bentrok. Kami juga butuh mencatat Rating mereka (bisa koma, misal 4.5). Terakhir, status mereka bisa 'Aktif' atau 'Suspend'."

  • Rating: Angka pecahan/desimal.

Kasus 2: Marketplace (Tabel Dompet User)

"Kami sedang membangun fitur E-Wallet bernama dompet_user. Tabel ini menyimpan saldo setiap user. Ingat, ini masalah uang, jadi angkanya harus sangat presisi (sampai 2 desimal) dan tidak boleh ada pembulatan error. Setiap dompet juga punya PIN keamanan (6 digit angka saja). Saldo awal user baru biasanya Rp 0."

ðŸ“‹ Analisa Kebutuhan:

  • ID Dompet: Primary Key.
  • ID User: Angka (Milik siapa dompet ini?).
  • Saldo: Angka Uang (Presisi tinggi, 2 angka di belakang koma). Default 0.
  • PIN: String angka (panjang tetap 6 karakter).
Tips: Uang jangan pakai FLOAT, pakailah DECIMAL(15, 2). PIN gunakan CHAR(6) atau VARCHAR(6).

Kasus 3: Blog Berita (Tabel Artikel)

"Tolong buatkan tabel artikel. Judul beritanya kadang panjang. Isi beritanya bisa sangat panjang (ribuan kata). Kita juga perlu mencatat kapan artikel ini dibuat. Kalau penulis lupa memasukkan waktu, sistem harus otomatis mencatat waktu saat ini."

ðŸ“‹ Analisa Kebutuhan:

  • ID: Primary Key.
  • Judul: Teks (agak panjang).
  • Isi Konten: Teks Sangat Panjang.
  • Tanggal Dibuat: Waktu lengkap. Default: Sekarang (Otomatis).
Tips: Isi konten gunakan TEXT atau LONGTEXT. Otomatis waktu gunakan DEFAULT CURRENT_TIMESTAMP.
